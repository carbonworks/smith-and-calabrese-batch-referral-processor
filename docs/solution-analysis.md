Tech stack: Google Workspace (paid), Microsoft 365 (paid), Google Sheets, Google Voice, Microsoft Word. All referrals through SSA's ELRQ system. Payment from state via MD Comptroller portal. This is an administrative government referral, not a doctor-to-doctor clinical referral. The referral forms are government-generated from a structured web system — likely standardized and consistent, which is favorable for automation.

## Problem Area 1: Invoice-to-Provider Payment Matching

### What's happening today

S&C is a Consultative Examination (CE) provider contracted by Maryland DDS to perform disability evaluations for SSA. DDS sends referrals through the SSA ELRQ system (https://secure.ssa.gov/apps9/EREApps/AER/ELRQ02View). Each referral is a government purchase order — DDS is buying an evaluation. The referral carries a PO# that should link it to its payment, but this key is buried in a downloadable PDF and not routinely used.

**Two data sources from SSA**:
1. **ELRQ web view** — displays referral data on-screen. Admin reads this and manually types it into Google Sheets. Open question: is PO# visible here?
2. **Downloadable PDF referral form** — government-generated CE referral containing PO#, claimant name, exam type, provider, etc. Standardized format (generated by a structured government system, not a freeform clinical document).

**Payments** come separately through the MD Comptroller portal (https://interactive.marylandtaxes.gov/extranet/gad/osvpi/User/main.asp).

**Current workflow**:
1. Admin reads referral data from ELRQ web view and types it into Google Sheet (~40-50 referrals/week)
2. Admin downloads payment data from MD Comptroller portal
3. Admin manually matches each payment to a provider using fuzzy criteria (client name + invoice amount + test type) — because PO# isn't in the Sheet
4. Ambiguous matches get escalated to Tyler

**Two bottlenecks** (relative time cost not yet confirmed — need to verify with Tyler):
1. **Manual transcription**: Reading from screen, typing into Sheet, 40-50 times per week. We believe this is the dominant time cost, but haven't confirmed.
2. **No match key**: PO# exists in the PDF but isn't extracted, so matching is slow and error-prone. If matching involves significant back-and-forth (ambiguous names, escalation to Tyler), this could rival or exceed the transcription time.

### What the solution actually needs to do

The technical work here is straightforward:
- **Parsing a web page**: Trivial. Walk the DOM with CSS selectors, extract text from table cells.
- **Parsing a PDF**: Trivial (assuming structured text, which is near-certain for government-generated PDFs).

The interesting questions are about the **UI** and the **spreadsheet**:

**UI question**: What form does the tool take that makes the admin's workflow better? A Chrome extension is the natural answer — the admin is already on the ELRQ page. Click a button, data gets extracted and formatted as CSV on the clipboard (or displayed in the extension popup), admin pastes into their Sheet. No Sheets API, no OAuth, no new login. The extension could also handle PDF parsing if we add a drag-and-drop or file picker to the popup.

**The spreadsheet is the real risk.** This Sheet is the practice's operational hub — referral tracking, payment reconciliation, and reminder sending all flow through it. Any solution that programmatically writes to this Sheet introduces risk to the business. A corrupted or mis-formatted entry could break matching, cause missed reminders, or create accounting errors.

The clipboard/paste approach mitigates this: the admin sees what they're pasting and controls where it goes. But even manual pasting has risk if the data doesn't match the Sheet's expected format or structure.

**The key question about the Sheet**:
- If each referral is a single row with simple columns (name, provider, date, type, location, amount) — and it appears this is the case — then the solution is trivial. Extract from web/PDF, format as CSV rows, paste. The admin validates visually.
- If the Sheet has complex structure (multi-row entries, cross-references between sheets, formulas that depend on specific cell positions, merged cells) — then we need to understand the structure deeply before building anything. The validation and fault tolerance requirements escalate quickly, and the effort to build a reliable tool may prove impractical relative to the manual process it replaces.

**This needs to be confirmed with Tyler before committing to build.**

### Distribution: 6 offices

**Working assumption**: Front desk / admin staff at each office handle the ELRQ data entry and reminders — not clinicians, not a central team. This is the standard model for multi-location practices (front desk has the non-clinical time between patients). Needs verification with Tyler.

**If distributed (likely)**: The Chrome extension needs to be installed on machines at all 6 offices. This means:

- **Distribution mechanism**: Start with **unpacked extension** (developer mode) — share a zip, each office loads it in `chrome://extensions`. No sign-up, no review process, instant deployment. If the developer mode nag banner becomes annoying, upgrade to Chrome Web Store unlisted ($5 one-time fee, adds auto-updates). Google Workspace admin push is also an option (they have paid Workspace) but requires Chrome Browser Cloud Management setup on each machine.
- **Training**: 6+ front desk staff need to learn the workflow. CW provides training materials (written guide + short video walkthrough). Tyler or office manager distributes and handles questions.
- **Consistency**: All offices need to use the same Sheet structure (or the extension needs to handle per-office Sheets). The screen share should clarify whether it's one shared Sheet or per-office.
- **Support/updates**: If a selector breaks (SSA updates the page), share an updated zip (or push via Chrome Web Store if upgraded). Chrome Web Store auto-updates are the main reason to eventually upgrade from unpacked distribution.

**If centralized** (less likely): Extension installs on 1-2 machines. Distribution is a non-issue. Training is minimal.

**Impact on scope**: Distribution adds modest effort — the extension itself is the same code regardless. The additional work is packaging (zip + install instructions, ~1 hour) and training materials (~2-3 hours). This is included in the estimate below.

### Risks

| Risk | Severity | What determines it |
|---|---|---|
| **Sheet structure is complex** (multi-row, cross-sheet references, fragile formulas) | **High** — could make the project impractical | Need to see the actual Sheet |
| PO# is not in the ELRQ web view | Medium — requires adding PDF parsing, but still straightforward | Ask Tyler; or inspect during screen share |
| ELRQ page structure is unusual (iframes, dynamic rendering) | Low — gov apps are typically server-rendered HTML | Inspect during development |
| SSA redesigns ELRQ interface | Low — gov sites change infrequently | Pin selectors to stable attributes; push updated zip (or auto-update via Web Store if upgraded) |
| Data pasted into wrong Sheet location | Medium — mitigated by clipboard approach (admin controls paste) | Clear column-mapping in training materials |
| Front desk staff inconsistency across offices | Medium — 6 people doing the same workflow could drift | Standardized training materials; 15/30-day follow-ups catch drift early |
| Emailed straggler referrals in different format | Low impact — ~5% of volume | Handle manually or build a separate handler later |

### Recommendation

**Build a Chrome extension** that extracts referral data from the ELRQ web view → CSV to clipboard + preview in extension UI. Admin pastes into their Sheet. If PO# isn't in the web view, add a PDF parser (drag-drop in the extension popup or a simple companion tool). Distribute as unpacked extension (zip file) for initial deployment; upgrade to Chrome Web Store unlisted ($5) if auto-updates are needed.

**Before building**: We need from Tyler:
1. **Where does the time go — transcription or matching?** (determines whether the extension scraper or PO# extraction is the higher-value piece)
2. **Is PO# visible in the ELRQ web view?** (determines whether PDF parsing is needed)
3. **What does the Google Sheet look like?** (the go/no-go gate — if entries are single-row, we're good; if the structure is complex, we need to reassess scope)
4. **Who does the ELRQ data entry — front desk at each office or a central person?** (confirms distribution needs)

A screen share where Tyler walks through the ELRQ page and their Sheet would answer all four in 15-20 minutes.

**Estimated effort**: 12-18 hours ($900-1,350 at $75/hr) for the extension including training materials. Add 5-10 hours ($375-750) if PDF parsing is needed. Pays for itself in under a month at 40-50 referrals/week.

**Deliverables**:
1. Chrome extension: ELRQ data → CSV clipboard + preview UI (+ PDF parsing if needed)
2. Distribution package (zip for unpacked install; Chrome Web Store upgrade optional)
3. Match formula/script for the Sheet (links Comptroller payments to providers)
4. Training materials (written guide + walkthrough) for front desk staff
5. 15 and 30-day follow-ups

**Ongoing support**:
- **90-day warranty**: Included in project price. Covers bug fixes, selector updates if SSA changes the ELRQ page, and minor adjustments based on real-world usage feedback. Starts at delivery.
- **After warranty**: Break/fix at hourly rate ($75/hr first-engagement pricing). Tyler contacts Randall when something breaks; Randall diagnoses and fixes, bills for time spent.
- **Diagnostics built in**: Extension detects when selectors fail and prompts the user to send a screenshot or page snapshot to Randall for debugging. Randall does not need (or have) credentials to the government portals.
- **Retainer option later**: If the engagement grows to include reminders and Word reports, ongoing support can bundle into a technology retainer ($250-500/mo) covering all tools.

## Problem Area 2: Automated Text Reminders

### Phase 1: Problem Diagnosis

**Current Workflow**:

| Step | Action | Who | Tool | Time Est. |
|------|--------|-----|------|-----------|
| 1 | Look up appointment details | Admin | Google Sheet | 1-2 min |
| 2 | Open Google Voice | Admin | Google Voice | 30 sec |
| 3 | Select location-specific template | Admin | Template doc | 30 sec |
| 4 | Customize message with patient details | Admin | Manual | 1 min |
| 5 | Send text | Admin | Google Voice | 30 sec |

**At 40-50 referrals/week**: 3-5 min/reminder × 40-50 = 120-250 min/week = **2-4 hours/week**. Plus potential for missed reminders, wrong location info, and human error.

**IS/IS NOT Diagnosis**:

| Dimension | IS | IS NOT |
|---|---|---|
| **WHAT** | Manual, repetitive message composition and sending | A complex messaging workflow — the template is simple |
| **WHERE** | Every appointment, every office location | Only some appointments or locations |
| **WHEN** | Before each appointment (likely 1-2 days prior) | Triggered by some other event |
| **EXTENT** | 40-50/week, 6 locations, each with different address/directions | A handful per week |

**Root Cause**: Google Voice has **no API** — no programmatic access to send SMS. Every reminder must be composed and sent by hand. The data (appointment info) is already in Sheets; the bottleneck is the sending mechanism.

**Classification**: **Solution limitation** — Google Voice simply cannot be automated. The problem disappears with a programmable SMS service.

**Future State**: Reminders sent automatically based on appointment data in Google Sheets. Location-specific templates applied automatically. Admin reviews/approves a batch rather than composing individually.

## Problem Area 3: LLM Tools for Microsoft Word Reports

### Phase 1: Problem Diagnosis

**Current Workflow**:

| Step | Action | Who | Tool | Time Est. |
|------|--------|-----|------|-----------|
| 1 | Administer test battery | Clinician | Various instruments | 1-6 hrs |
| 2 | Score tests | Clinician | Manual/scoring software | 15-30 min |
| 3 | Open Word template | Clinician | Microsoft Word | 1 min |
| 4 | Insert test scores into template tables | Clinician | Manual typing | 10-15 min |
| 5 | Write score descriptions/interpretations | Clinician | Manual writing | 10-20 min |
| 6 | Write narrative sections | Clinician | Manual writing | 20-40 min |
| 7 | Review and finalize | Clinician/Tyler | Word | 10-15 min |

**At 40-50 reports/week**: Steps 4-6 = 40-75 min/report × 40-50/week = 27-62 hours/week of clinician time on report writing across the practice.

**IS/IS NOT Diagnosis**:

| Dimension | IS | IS NOT |
|---|---|---|
| **WHAT** | Manual data entry (scores, descriptions) + manual narrative writing in Word | A documentation problem — they have templates; the bottleneck is populating them |
| **WHERE** | In Word, during report writing after assessment | During the assessment itself or during scoring |
| **WHEN** | Every report, every assessment | Some reports but not others |
| **EXTENT** | 40-50 reports/week; significant clinician hours | A few per week |

**Root Cause**: Two distinct sub-problems:
1. **Structured data entry**: Test scores, percentiles, and standard descriptions are looked up from reference tables and manually typed into Word tables. This is mechanical — a computer could do it.
2. **Narrative writing**: Clinical interpretation and narrative sections require professional judgment but follow patterns. AI could draft, clinician reviews.

**Classification**:
- Sub-problem 1 (scores/tables): **Solution limitation** — Word has no built-in score lookup. This is a data automation problem, not an AI problem.
- Sub-problem 2 (narratives): **Both** — No AI tool exists for this specific report type (solution), and clinicians would need to trust and adopt AI-assisted writing (enterprise).

**Future State**: Score/description insertion is automated from test results. Narrative drafting is AI-assisted. Clinician time per report reduced from 40-75 min to 15-30 min.
